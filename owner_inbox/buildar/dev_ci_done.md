## Decisions Made

### pnpm Version: `9` (not `11.1.2`)

The spec referenced `pnpm@11.1.2` as a starting point and instructed me to verify the exact version from `pnpm-lock.yaml`. Upon inspection:

- `pnpm-lock.yaml` line 1: `lockfileVersion: '9.0'`
- `package.json`: no `packageManager` field
- `.github/workflows/README.md` (pre-existing): explicitly references "pnpm 9"

Lockfile format `9.0` is the format generated by pnpm v9.x. Using pnpm 11.1.2 to install a lockfile generated by pnpm 9 risks format mismatches that would cause `--frozen-lockfile` to fail in CI. I pinned to `pnpm@9` (latest patch in the v9 series) to match what generated the lockfile. This is the correct safe choice.

**If the team upgrades to pnpm 11**, the right migration path is: upgrade locally first, run `pnpm install` to regenerate the lockfile, commit both, then update the CI version.

### Install method: `npm install -g pnpm@9`

The spec offered two options:
1. `npm install -g pnpm@11.1.2`
2. `corepack enable && corepack prepare pnpm@11.1.2 --activate`

I used option 1 (`npm install -g pnpm@9`) because:
- Corepack on `ubuntu-latest` requires the `packageManager` field in `package.json` to be set — this repo does not have it, so corepack would install the wrong version or fail silently.
- `npm install -g` is unconditional and predictable.
- pnpm must be installed **before** `actions/setup-node` runs so that the `cache: 'pnpm'` option can locate the pnpm store for caching.

### Step order adjustment

The spec listed: checkout → setup-node → install pnpm → install deps → typecheck → lint → build.

I moved "Install pnpm" **before** "Setup Node" because `actions/setup-node@v4` with `cache: 'pnpm'` probes for the pnpm binary and store path at setup time. If pnpm is not yet installed, the cache step fails silently or errors. The correct sequence is: checkout → install pnpm → setup-node (with cache) → install deps → typecheck → lint → build.

### Cache key

`cache-dependency-path: pnpm-lock.yaml` is passed explicitly to `actions/setup-node`. This ensures the pnpm store cache key is derived from `hashFiles('pnpm-lock.yaml')` as required by the spec.

### Pre-existing ci.yml

A previous `ci.yml` existed (multi-job design: install / lint / typecheck / test / migration-validate). It did not match the spec (single `ci` job, no test or migration jobs in Phase A). It was overwritten per the task spec.

- **`pnpm -r` flags:** Using `-r` (recursive) on typecheck, lint, build means each workspace's script is invoked. Workspaces without the script are skipped by pnpm without error. This is safe and forward-compatible.
- **No test step:** Excluded per spec (no test scripts yet).
- **No migration-validate step:** Excluded per spec (Phase B).
- **No matrix builds, path filters, or deploy jobs:** Excluded per spec.
